# ADR-001：ADE 首期实施基线

- 状态：已接受
- 日期：2026-07-10
- 适用范围：PoC 与 MVP

## 背景

ADE 要求多个不同 Agent 使用同一个项目目录，同时又要求修改可审核、可拒绝、可恢复。终端型 Agent 的能力并不一致，Office 的视觉版面与 OOXML 结构也无法天然一一对应。如果不先固定执行契约，UI、Agent 适配器、权限层和版本系统会产生互相矛盾的假设。

## 决策一：首期采用直接事务模式

MVP 中 Agent 的 `cwd` 使用 Windows 项目真实目录。任务开始前创建文件清单和内容快照，运行中监测全部文件变化，任务完成后形成 `ChangeSet`。

- 接受：保留 Agent 已写入的变化，并生成接受版本。
- 拒绝：使用快照恢复任务涉及的变化。
- 部分接受：文本文件应用选择后的补丁；二进制文件首期只支持整文件接受或拒绝。
- 外部修改：若任务运行期间检测到非任务进程造成的冲突，停止自动恢复并要求用户处理。

该模式不声称“接受前修改没有落地”。阶段 2 再实现隔离候选模式，用于并行 Agent、方案对比和真正的合并后生效。

## 决策二：多个 Agent 可配置，MVP 串行执行

一个项目可以注册多个 Agent，用户可将不同任务派给不同 Agent，但同一项目首期只允许一个写任务运行。只读任务可以并行，前提是 Provider 能被可靠标记为只读。

阶段 2 使用隔离候选工作区实现并行写任务，不依赖应用级文件锁保障安全。

## 决策三：Agent Provider 采用能力分级

| 等级 | 输入输出 | 结构化事件 | 权限拦截 | 会话恢复 |
|---|---|---|---|---|
| SDK | 原生 API | 完整 | 可用 | 可用 |
| RPC | JSONL/RPC | 大部分 | 视协议而定 | 视协议而定 |
| Enhanced PTY | ConPTY + 已知输出契约 | 部分 | 通常不可用 | 视 CLI 而定 |
| Raw PTY | 原始终端 | 不保证 | 不可用 | 不保证 |

Core 只依赖必选能力。UI 根据能力矩阵显示或隐藏暂停、恢复、工具详情和权限拦截功能。

MCP 作为工具连接协议进入 `tool-connector`，不承担 Agent 进程和会话生命周期。

## 决策四：内部路径全部项目相对化

本地 Agent 使用 `hostRoot`，WSL/Docker/SSH 使用 `runtimeRoot`。任务、批注、产物和审计记录保存项目相对路径，由 `WorkspaceBinding` 在运行时做双向转换。

## 决策五：Office 使用三层文档模型

1. `StructureModel`：OOXML 元素、文本、表格、形状和公式。
2. `VisualSnapshot`：渲染后的页面、幻灯片或工作表视图。
3. `RenderMap`：结构元素和视觉坐标之间的映射及置信度。

RenderMap 不能可靠建立时，批注降级为视觉区域、截图、引文和人工确认，不伪造精确元素定位。

## 决策六：安全控制分层

- 调度控制：任务队列、写任务串行化、应用级文件锁。
- 检测控制：文件监视、越界变更告警、命令与事件审计。
- 恢复控制：内容快照、原子写入、接受/拒绝。
- 强制控制：后续使用隔离工作区、ACL、AppContainer 或文件代理。

Job Objects 用于收拢子进程和限制资源，ConPTY 用于终端交互；二者不被描述为路径级沙箱。

## 首个垂直切片

首个可验收切片只包含：

1. 打开一个本地项目。
2. 浏览文件树并打开 Markdown。
3. 在渲染视图中选择文本并创建批注。
4. 选择一个模拟 Agent，生成结构化任务包。
5. 展示任务状态、上下文和预期修改范围。

随后用真实 CLI Provider 替换模拟 Agent，再加入快照、文件监视和 diff。
